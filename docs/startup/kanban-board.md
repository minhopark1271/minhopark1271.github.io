## 칸반이란

칸반(看板, Kanban)은 일본어로 **"시각 신호판"**을 의미합니다. 看(보다) + 板(판)의 합성어로, 작업 상태를 시각적으로 표현하는 보드 기반의 워크플로 관리 시스템입니다.

핵심 아이디어는 단순합니다:

> **"지금 진행 중인 일을 눈에 보이게 만들고, 한 번에 처리하는 양을 제한하라."**

칸반보드는 보통 **To Do → In Progress → Done** 같은 열(column)로 구성되며, 각 작업을 카드로 표현하여 왼쪽에서 오른쪽으로 이동시킵니다.

```
┌──────────┬──────────┬──────────┐
│  To Do   │  Doing   │   Done   │
│          │ (WIP: 3) │          │
├──────────┼──────────┼──────────┤
│ [Task A] │ [Task D] │ [Task F] │
│ [Task B] │ [Task E] │ [Task G] │
│ [Task C] │          │          │
└──────────┴──────────┴──────────┘
```

## 역사

### 1940~50년대: 도요타 생산시스템 (TPS)

칸반의 기원은 **타이이치 오노(大野耐一, Taiichi Ohno)**가 설계한 도요타 생산시스템(Toyota Production System)입니다.

오노는 미국 슈퍼마켓에서 영감을 얻었습니다. 슈퍼마켓은 재고를 미리 가득 채우는 것이 아니라, **선반이 비어갈 때만 보충**합니다. 이 관찰에서 **적시 생산(Just-in-Time, JIT)** 개념이 탄생했습니다.

- **Push → Pull 전환**: 수요 예측 기반 생산(push)에서, 실제 소비에 의해 생산이 촉발되는 pull 방식으로 전환
- **칸반 카드**: 부품 소비 시 칸반 카드를 이전 공정으로 보내 보충 신호를 전달
- **재고 최소화**: 과잉 생산을 방지하고 낭비(무다, muda)를 제거

### 2004~2010년: 소프트웨어 개발로의 전환

**David J. Anderson**이 칸반을 지식 노동(knowledge work)에 적용한 핵심 인물입니다.

| 시기 | 사건 |
|------|------|
| 2004 | Microsoft XIT 부서에서 최초의 소프트웨어 칸반 시스템 적용 |
| 2006 | Corbis에서 결과 검증 및 확산 |
| 2007 | 칸반 메서드의 6가지 실천법(practices) 정의 |
| 2010 | Anderson의 저서 *"Kanban: Successful Evolutionary Change"* 출간 |
| 2011~ | 대기업 본격 도입, 애자일 생태계 주류 편입 |

Anderson은 **제약이론(TOC)**, **린 사고(Lean Thinking)**, **W. Edwards Deming의 시스템 사고**를 결합하여 소프트웨어 칸반 메서드를 체계화했습니다.

## 핵심 원칙

칸반 메서드는 **4가지 기본 원칙**과 **6가지 핵심 실천법**으로 구성됩니다.

### 4가지 원칙

1. **현재 프로세스에서 시작하라** — 기존 역할과 프로세스를 존중하며 점진적으로 개선
2. **점진적·진화적 변화를 추구하라** — 급진적 혁신이 아닌 지속적 개선
3. **현재 역할·책임·직함을 존중하라** — 조직 변경 없이 도입 가능
4. **모든 수준에서 리더십을 장려하라** — 팀원 누구나 개선을 제안

### 6가지 실천법

| 실천법 | 설명 |
|--------|------|
| **워크플로 시각화** | 보드와 카드로 작업 흐름을 눈에 보이게 표현 |
| **WIP 제한** | 동시 진행 작업 수를 제한하여 집중도 향상 |
| **흐름 관리** | 병목 구간 식별 및 해소, 리드타임 단축 |
| **정책 명시화** | 완료 조건, 우선순위 기준 등을 명문화 |
| **피드백 루프 구현** | 스탠드업, 리뷰 등 정기적 피드백 회의 운영 |
| **협력적 개선** | 실험과 데이터 기반의 지속적 프로세스 개선 |

## 사용법

### 기본 보드 설계

가장 단순한 형태는 3열 보드이지만, 실무에서는 팀 상황에 맞게 열을 확장합니다.

```
┌─────────┬──────────┬──────────┬───────────┬─────────┐
│ Backlog │ Analysis │  Dev     │  Review   │  Done   │
│         │ (WIP: 2) │ (WIP: 3) │  (WIP: 2) │         │
└─────────┴──────────┴──────────┴───────────┴─────────┘
```

### WIP 제한 설정

WIP(Work In Progress) 제한은 칸반의 가장 강력한 도구입니다.

- **설정 기준**: 팀원 수의 1~1.5배로 시작 (예: 개발자 4명 → WIP 4~6)
- **효과**: 멀티태스킹 감소 → 컨텍스트 스위칭 비용 절감 → 완료율 향상
- **조정**: 병목이 발견되면 WIP를 줄여 문제를 표면화

### 칸반 카드 구성

각 카드에 포함할 정보:

- **제목**: 작업 내용 요약
- **담당자**: 현재 작업자
- **유형 표시**: 버그 수정, 기능 개발, 리서치 등
- **기한**: 있는 경우 표시
- **블로커 표시**: 진행을 막는 요소가 있으면 표시

### 주요 메트릭

| 메트릭 | 정의 | 활용 |
|--------|------|------|
| **리드타임** | 요청 → 완료까지 소요 시간 | 고객 관점의 배포 속도 측정 |
| **사이클타임** | 작업 시작 → 완료까지 소요 시간 | 팀 내부 효율성 측정 |
| **처리량** | 단위 시간당 완료된 작업 수 | 팀 생산성 추적 |
| **WIP 수** | 현재 진행 중인 작업 수 | 과부하 여부 판단 |

### 대표 도구

- **Trello** — 직관적 UI, 소규모 팀에 적합
- **Jira** — 스크럼/칸반 모두 지원, 대규모 팀용
- **Notion** — 문서 + 보드 통합, 스타트업에서 인기
- **GitHub Projects** — 개발 워크플로와 자연스럽게 연동
- **Linear** — 빠른 속도와 깔끔한 UX, 엔지니어링 팀 선호

## 철학적 배경

칸반은 단일 이론이 아니라 여러 철학과 관리 이론이 융합된 결과물입니다.

### 린 사고 (Lean Thinking)

도요타 생산시스템에서 비롯된 철학으로, **7가지 낭비(무다) 제거**를 핵심으로 합니다.

1. 과잉 생산 (Overproduction)
2. 대기 (Waiting)
3. 불필요한 운반 (Transportation)
4. 과잉 처리 (Over-processing)
5. 재고 (Inventory)
6. 불필요한 동작 (Motion)
7. 결함 (Defects)

칸반의 WIP 제한과 Pull 시스템은 린의 낭비 제거 원칙을 직접 구현한 것입니다.

### 제약이론 (Theory of Constraints, TOC)

**엘리야후 골드랫(Eliyahu Goldratt)**이 1984년 저서 *The Goal*에서 제시한 이론입니다.

> **"시스템 전체의 성과는 가장 약한 고리(제약)에 의해 결정된다."**

TOC의 5단계 집중 개선:

1. 제약 식별 (Identify)
2. 제약 최대 활용 (Exploit)
3. 비제약을 제약에 종속 (Subordinate)
4. 제약 역량 강화 (Elevate)
5. 반복 (Repeat)

칸반에서 WIP 제한은 병목을 의도적으로 표면화시키는 장치로, TOC의 "제약 식별" 단계와 직결됩니다.

### 데밍의 시스템 사고

**W. Edwards Deming**의 관리 철학에서 핵심 개념을 차용합니다.

- **PDCA 사이클** (Plan-Do-Check-Act): 칸반의 지속적 개선(Kaizen)과 일맥상통
- **변동의 이해**: 프로세스 변동을 데이터로 측정하고 관리
- **시스템 사고**: 부분 최적화가 아닌 전체 시스템 최적화 추구

### 카이젠 (改善, Kaizen)

"개선"을 의미하는 일본어로, **작지만 지속적인 개선**을 강조합니다. 칸반의 "점진적·진화적 변화" 원칙이 바로 카이젠 철학의 반영입니다.

## 장단점

### 장점

| 장점 | 설명 |
|------|------|
| **낮은 도입 장벽** | 기존 프로세스 위에 바로 적용 가능, 조직 변경 불필요 |
| **시각적 직관성** | 전체 작업 현황을 한눈에 파악 |
| **유연한 우선순위** | 스프린트 없이 언제든 우선순위 변경 가능 |
| **병목 가시화** | WIP 제한으로 문제 지점이 자연스럽게 드러남 |
| **지속적 배포** | 완료되는 즉시 배포, 릴리즈 주기에 구애받지 않음 |
| **팀 자율성** | 역할이 고정되지 않아 팀원이 유연하게 대응 |

### 단점

| 단점 | 설명 |
|------|------|
| **구조 부족** | 역할, 시간 제약이 없어 규율이 느슨해질 수 있음 |
| **예측 어려움** | 스프린트가 없어 "언제 완료되는가"에 답하기 어려움 |
| **과도한 WIP** | WIP 제한을 준수하지 않으면 효과가 크게 감소 |
| **복잡한 프로젝트에 한계** | 대규모·장기 프로젝트의 전체 로드맵 관리에 취약 |
| **성숙도 의존** | 팀의 자기 관리 능력이 낮으면 혼란 가능 |

## 대체 방법론 비교

### 스크럼 (Scrum)

| 항목 | 칸반 | 스크럼 |
|------|------|--------|
| **주기** | 연속적 흐름 | 1~4주 고정 스프린트 |
| **역할** | 정해진 역할 없음 | PO, SM, Dev Team |
| **변경** | 언제든 우선순위 변경 가능 | 스프린트 내 변경 지양 |
| **계획** | 필요 시 수시 | 스프린트 플래닝 |
| **메트릭** | 리드타임, 사이클타임 | 벨로시티 |
| **적합 상황** | 운영, 유지보수, 변동 큰 업무 | 신규 개발, 명확한 목표 |

### 스크럼반 (Scrumban)

스크럼과 칸반의 하이브리드 방식입니다.

- 스크럼의 **스프린트 구조와 회고** 채택
- 칸반의 **WIP 제한과 시각적 보드** 결합
- 스프린트 내에서 유연한 작업 추가/변경 허용

### 워터폴 (Waterfall)

전통적인 순차적 개발 모델로, 요구사항 → 설계 → 구현 → 테스트 → 배포를 단계별로 진행합니다. 요구사항이 초기에 확정되고 변경이 거의 없는 프로젝트에 적합합니다.

### Shape Up (Basecamp)

Basecamp가 제안한 방법론으로, **6주 사이클 + 2주 쿨다운**으로 운영됩니다. 스크럼보다 긴 호흡, 칸반보다 명확한 기한을 제공합니다.

### XP (Extreme Programming)

기술적 실천법(페어 프로그래밍, TDD, CI/CD 등)에 중점을 둔 방법론입니다. 칸반/스크럼과 조합하여 사용하는 경우가 많습니다.

## 도입 시 팁

1. **작게 시작하라** — 한 팀, 하나의 보드로 시작. 전사 도입은 성공 사례 확보 후
2. **WIP 제한을 반드시 지켜라** — WIP 제한 없는 칸반은 그냥 To-Do 리스트
3. **정기 회고를 하라** — 칸반은 역할/회의를 강제하지 않지만, 피드백 루프 없이는 개선이 없음
4. **데이터를 측정하라** — 리드타임, 사이클타임을 추적해야 개선 방향이 보임
5. **완벽한 보드를 만들지 마라** — 보드는 현실을 반영해야지, 이상을 표현하는 게 아님

---

## 요약

칸반은 **"일을 눈에 보이게 만들고, 한 번에 하는 양을 줄여라"**는 단순한 원칙 위에 세워진 방법론입니다.

도요타 공장에서 시작된 이 아이디어는 린 사고, 제약이론, 데밍의 시스템 사고를 흡수하며 소프트웨어 개발과 지식 노동 전반으로 확산되었습니다. 스크럼처럼 구조를 강제하지 않기에 도입이 쉽지만, WIP 제한과 지속적 개선이라는 핵심을 놓치면 단순한 작업 목록으로 전락합니다.

팀 상황에 맞게 열을 설계하고, WIP 제한을 준수하며, 데이터 기반으로 병목을 개선하는 것이 칸반 활용의 핵심입니다.
