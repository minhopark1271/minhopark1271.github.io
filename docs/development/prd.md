# PRD(Product Requirement Document)
{:.no_toc}

제품을 만들기 전에 **무엇을, 왜, 어떻게** 만들 것인지를 정의하는 문서입니다. AI 코딩 에이전트의 등장으로 PRD의 역할이 '팀 커뮤니케이션 도구'에서 **'AI에게 건네는 실행 가능한 설계도'**로 확장되고 있습니다.

## 1. PRD란 무엇인가

**PRD(Product Requirement Document, 제품 요구사항 문서)**는 제품의 목표, 주요 기능, 기술적 요구사항, 제약 조건 등을 체계적으로 기록한 문서입니다. 제품 관리자(PM)가 주도적으로 작성하며, 개발팀/디자인팀/QA팀 등 모든 이해관계자가 **동일한 목표를 향해 정렬**하도록 하는 것이 핵심 목적입니다.

### PRD가 답해야 하는 질문

| 질문 | 설명 |
|------|------|
| **What** | 어떤 제품/기능을 만드는가? |
| **Why** | 왜 만들어야 하는가? (비즈니스 가치) |
| **Who** | 타겟 사용자는 누구인가? |
| **How** | 어떻게 동작해야 하는가? (기능 요구사항) |
| **When** | 언제까지 완성해야 하는가? (마일스톤) |
| **What not** | 무엇을 하지 않을 것인가? (비목표) |

### PRD가 필요한 이유

Carnegie Mellon Software Engineering Institute의 연구에 따르면, 소프트웨어 개발 비용의 **60~80%가 수정(rework)**에 사용됩니다. 효과적인 요구사항 관리는 프로젝트 결함의 50~80%를 사전에 제거할 수 있습니다.

- **팀 정렬**: 모든 이해관계자가 같은 방향을 바라봄
- **범위 관리**: "이건 이번 버전에 안 들어갑니다"를 근거 있게 말할 수 있음
- **의사결정 기록**: 왜 그렇게 결정했는지 추적 가능
- **품질 기준선**: 무엇이 "완료"인지 명확히 정의

## 2. 유사 문서와의 비교

PRD와 혼동되는 문서들이 있습니다. 각각의 역할과 초점이 다릅니다.

| 문서 | 정식 명칭 | 핵심 질문 | 주요 독자 | 수준 |
|------|-----------|-----------|-----------|------|
| **BRD** | Business Requirements Document | **왜** 하는가? | 경영진, 사업부 | 고수준 (전략) |
| **PRD** | Product Requirement Document | **무엇을** 만드는가? | PM, 개발팀, 디자인팀 | 중수준 (제품) |
| **SRS** | Software Requirements Specification | **어떻게** 동작하는가? | 개발팀, QA | 저수준 (기술) |
| **FRD** | Functional Requirements Document | **어떤 기능**을 수행하는가? | 개발팀, QA | 저수준 (기능) |

**관계**: BRD → PRD → SRS/FRD 순서로 구체화됩니다. BRD가 "왜"를 정의하면, PRD가 "무엇을"로 구체화하고, SRS/FRD가 "어떻게"를 상세 기술합니다.

> 실무에서는 PRD가 SRS의 역할까지 포함하는 경우가 많습니다. 특히 소규모 팀이나 애자일 조직에서는 별도의 SRS를 작성하지 않고 PRD 하나로 통합하는 경향이 있습니다.

## 3. PRD의 핵심 구성 요소

### 3.1 메타데이터 (표지 정보)

```markdown
- 문서 제목
- 버전 (v1.0, v1.1, ...)
- 작성자 / 소유자
- 최종 수정일
- 상태 (Draft / In Review / Approved)
- 변경 이력 (날짜, 수정자, 내용)
```

변경 이력은 IEEE 표준에서도 권장하는 필수 항목입니다. "누가, 언제, 왜 변경했는지"를 추적합니다.

### 3.2 개요 (Overview)

프로젝트의 **목적과 배경**을 간결하게 설명합니다. 이 문서를 끝까지 읽지 않을 사람도 핵심을 파악할 수 있어야 합니다.

포함 내용:
- 해결하려는 문제 (Problem Statement)
- 제안하는 솔루션의 핵심 아이디어
- 프로젝트의 비즈니스 가치 또는 전략적 의미

### 3.3 목표 및 성공 지표 (Goals & Success Metrics)

**비즈니스 목표**와 이를 측정할 **정량적 지표(KPI)**를 정의합니다.

```markdown
목표: 사용자 온보딩 완료율 향상
성공 지표:
- 가입 후 7일 내 온보딩 완료율: 현재 45% → 목표 70%
- 온보딩 이탈 단계 감소: 현재 3단계 → 2단계 이하
- NPS 점수: 현재 32 → 목표 45
```

### 3.4 타겟 사용자 및 페르소나 (Target Users & Personas)

누구를 위한 제품인지 구체적으로 정의합니다.

```markdown
페르소나 1: "바쁜 개발자 민호"
- 역할: 백엔드 개발자 (3년차)
- 특성: CLI 선호, 문서보다 코드 우선
- 니즈: 반복 작업 자동화, 빠른 디버깅
- 페인 포인트: 수동 배포, 불완전한 테스트 커버리지
```

### 3.5 사용자 시나리오 / 유스케이스

페르소나가 **실제로 제품을 사용하는 흐름**을 스토리 형태로 기술합니다.

```markdown
시나리오: "민호가 새 API 엔드포인트를 추가한다"
1. 민호가 PRD를 작성하고 AI 에이전트에 전달
2. 에이전트가 기존 코드 구조를 분석
3. 에이전트가 구현 계획을 제시 → 민호가 검토/승인
4. 에이전트가 코드 생성 + 테스트 작성
5. 민호가 최종 리뷰 후 커밋
```

### 3.6 기능 요구사항 (Functional Requirements)

제품이 **수행해야 하는 기능**을 상세히 나열합니다. 우선순위(P0/P1/P2)를 반드시 지정합니다.

| 우선순위 | 기능 | 설명 | 수용 기준 |
|----------|------|------|-----------|
| P0 (필수) | 사용자 인증 | 이메일+비밀번호 로그인 | 유효한 자격증명으로 로그인 성공, 잘못된 정보 시 에러 표시 |
| P0 (필수) | 대시보드 | 핵심 지표 실시간 표시 | 5초 이내 데이터 로딩, 자동 갱신 |
| P1 (중요) | 알림 설정 | 사용자별 알림 규칙 설정 | 최소 3종류 알림 채널 지원 |
| P2 (선택) | 다크 모드 | UI 테마 전환 | 모든 페이지에서 일관된 테마 적용 |

> **P0**: 없으면 출시 불가 / **P1**: 출시 후 빠른 시일 내 필요 / **P2**: 있으면 좋지만 없어도 출시 가능

### 3.7 비기능 요구사항 (Non-Functional Requirements)

| 항목 | 요구사항 |
|------|----------|
| **성능** | API 응답 시간 200ms 이내 (p95) |
| **가용성** | 99.9% uptime |
| **확장성** | 동시 접속 1,000명 처리 |
| **보안** | OWASP Top 10 대응, 데이터 암호화 |
| **접근성** | WCAG 2.1 AA 준수 |
| **호환성** | Chrome, Safari, Firefox 최신 2개 버전 |

### 3.8 비목표 및 제외 범위 (Non-Goals & Out of Scope)

**의도적으로 하지 않을 것**을 명시합니다. 이것이 없으면 범위가 끝없이 확장(scope creep)됩니다.

```markdown
비목표 (이번 버전에서 제외):
- 모바일 네이티브 앱 개발
- 다국어 지원 (한국어만 지원)
- 소셜 로그인 (이메일 로그인만)
- 오프라인 모드
```

### 3.9 기술적 제약 조건 및 의존성

```markdown
제약 조건:
- Python 3.11 환경 (3.12+ 미지원)
- MySQL 8.0 데이터베이스
- TensorFlow 2.16.2 (2.17+ 호환성 이슈)

외부 의존성:
- Binance API (실시간 시세 데이터)
- OpenAI API (텍스트 분석)
```

### 3.10 일정 및 마일스톤

```markdown
Phase 1 (2주): 데이터 수집 파이프라인 구축
Phase 2 (2주): 기술 지표 계산 엔진 구현
Phase 3 (3주): 예측 모델 학습 및 검증
Phase 4 (1주): API 서버 및 대시보드 배포
```

### 3.11 미해결 사항 (Open Questions)

아직 결정되지 않은 항목을 투명하게 기록합니다. TBD(To Be Determined)로 표기합니다.

```markdown
미해결:
- [ ] 알림 전송 주기: 실시간 vs 5분 배치?
- [ ] 데이터 보존 기간: 90일 vs 1년?
- [ ] 모니터링 도구 선정: Grafana vs Datadog?
```

## 4. PRD 유형별 특성

프로젝트 규모와 조직 문화에 따라 PRD의 형태가 달라집니다.

### One-Pager PRD (간결형)

- 소규모 기능, 빠른 이터레이션에 적합
- 핵심 요소만 포함: 문제, 솔루션, 성공 지표, 핵심 사용자 스토리
- 1~2페이지 분량

### Full PRD (표준형)

- 대규모 프로젝트, 다수 팀 협업에 적합
- 위 3장의 모든 구성 요소 포함
- 5~15페이지 분량

### Agile PRD (애자일형)

- 스프린트 기반 개발에 적합
- 사용자 스토리 + 수용 기준 중심
- 스프린트마다 업데이트되는 **살아있는 문서**
- 백로그 아이템과 직접 연결

## 5. PRD 작성 모범 사례

### 작성 원칙

1. **간결하게**: 팀이 실제로 읽을 길이로 제한합니다. 과도하게 긴 PRD는 읽히지 않습니다
2. **측정 가능하게**: "사용성을 개선한다" (X) → "온보딩 완료율을 70%로 올린다" (O)
3. **모호함 제거**: "빠른 응답" (X) → "200ms 이내 응답" (O)
4. **결정 중심**: 토론이 필요한 항목에 집중합니다. 자명한 내용은 줄입니다
5. **살아있는 문서**: 프로젝트 진행에 따라 지속적으로 업데이트합니다

### 검토 프로세스

```
PM 초안 작성
    → 디자인팀 리뷰 (UX 타당성)
    → 엔지니어링팀 리뷰 (기술 실현 가능성)
    → 이해관계자 최종 승인
    → 개발 착수
```

### 흔한 실수

| 실수 | 개선 방향 |
|------|-----------|
| 솔루션부터 정의 | 문제 정의를 먼저, 솔루션은 그 다음 |
| 비목표 미기재 | 하지 않을 것을 명시하여 범위 관리 |
| 성공 지표 없음 | 측정 가능한 KPI 반드시 포함 |
| 한 번 쓰고 방치 | 변경사항을 지속 반영 |
| 너무 상세한 기술 명세 | PRD는 "무엇을"에 집중, "어떻게"는 개발팀에 위임 |

## 6. 실전 PRD 예시

간단한 예시로 전체 구조를 확인합니다.

```markdown
# PRD: 트레이딩 봇 알림 시스템

## 메타데이터
- 버전: v1.0
- 작성자: 홍길동 (PM)
- 상태: In Review
- 최종 수정: 2026-02-13

## 1. 개요
트레이딩 봇의 예측 결과와 포지션 변화를 사용자에게 실시간으로 알려주는
알림 시스템을 구축합니다. 현재 사용자는 대시보드를 직접 확인해야만
상태 변화를 인지할 수 있어, 중요한 시그널을 놓치는 문제가 있습니다.

## 2. 목표 및 성공 지표
- 목표: 중요 시그널 놓침률 80% 감소
- KPI: 알림 수신 후 평균 반응 시간 5분 이내
- KPI: 알림 오탐률(false positive) 10% 이하

## 3. 타겟 사용자
- 개인 트레이더: 모바일로 시장을 모니터링하는 사용자
- 알고리즘 트레이더: 자동화 파이프라인에 알림을 통합하고 싶은 사용자

## 4. 기능 요구사항
| 우선순위 | 기능 | 수용 기준 |
|----------|------|-----------|
| P0 | Telegram 알림 | 예측 결과 변경 시 30초 이내 발송 |
| P0 | 알림 규칙 설정 | 최소 3가지 조건 조합 지원 |
| P1 | 알림 이력 조회 | 최근 7일 이력 대시보드 표시 |
| P2 | Slack 연동 | Webhook 기반 메시지 전송 |

## 5. 비기능 요구사항
- 알림 전송 지연: 30초 이내 (p99)
- 시스템 가용성: 99.5%
- 일일 최대 알림 수: 사용자당 100건 제한

## 6. 비목표
- 이메일 알림 (이번 버전 제외)
- 알림 내 직접 트레이딩 실행 기능
- 다국어 알림 메시지

## 7. 기술 제약
- 기존 FastAPI 서버에 엔드포인트 추가
- Telegram Bot API 사용
- MySQL에 알림 이력 저장

## 8. 일정
- Phase 1 (1주): Telegram 연동 + 기본 알림
- Phase 2 (1주): 알림 규칙 엔진 + 대시보드

## 9. 미해결 사항
- [ ] 알림 메시지 포맷 확정 (텍스트 vs 이미지 차트)
- [ ] 야간 알림 무음 설정 필요 여부
```

## 7. AI 시대의 PRD

### 전통적 PRD에서 AI 시대 PRD로

AI 코딩 에이전트(Claude Code, Cursor, Codex 등)의 등장으로 PRD의 역할이 진화하고 있습니다.

| 구분 | 전통적 PRD | AI 시대 PRD |
|------|-----------|-------------|
| **독자** | 사람 (개발자, 디자이너) | 사람 + AI 에이전트 |
| **목적** | 팀 간 의사소통, 정렬 | 의사소통 + AI 실행 지시 |
| **형식** | 자유 형식, 긴 산문 가능 | 구조화, 체크리스트, 명시적 경계 |
| **모호함** | 회의에서 구두로 해소 | AI가 추론할 수 없으므로 모호함 제거 필수 |
| **비목표** | 있으면 좋음 | **필수** (AI는 생략된 것을 추론 못함) |
| **업데이트** | 초기 작성 후 방치 경향 | 반복 실행마다 실시간 업데이트 |

핵심 변화는 **명시성(explicitness)**입니다. 사람은 "당연히 인증이 필요하겠지"라고 추론하지만, AI 에이전트는 PRD에 명시되지 않으면 인증을 추가할 수도, 안 할 수도 있습니다. 반대로 명시적으로 제외하지 않으면 불필요한 기능을 임의로 추가할 수도 있습니다.

### AI 코딩 에이전트에게 PRD가 중요한 이유

AI 코딩 에이전트는 스펙 문서를 기반으로 작동합니다. PRD가 곧 **AI에게 건네는 설계도**입니다.

1. **컨텍스트 제공**: AI가 전체 프로젝트의 맥락을 이해
2. **범위 제한**: 불필요한 코드 생성 방지
3. **품질 기준**: 테스트와 수용 기준을 사전 정의
4. **일관성 유지**: 여러 세션에 걸친 작업에서 동일한 방향 유지

---

## 8. AI 에이전트를 위한 PRD 작성법

AI 코딩 에이전트에게 효과적으로 작업을 지시하기 위한 PRD 작성 전략입니다.

### 8.1 6가지 필수 영역

Addy Osmani가 제안한 AI 에이전트 스펙의 6가지 핵심 영역:

| 영역 | 내용 | 예시 |
|------|------|------|
| **Commands** | 빌드, 테스트, 린트 등 실행 명령 | `pytest tests/`, `ruff check .` |
| **Testing** | 테스트 프레임워크와 커버리지 기대치 | pytest + 80% 커버리지 |
| **Project Structure** | 코드 구조 및 디렉토리 배치 | `src/` 소스, `tests/` 테스트 |
| **Code Style** | 코딩 컨벤션 (실제 코드 샘플 포함) | 라인 100자 제한, Black 포매터 |
| **Git Workflow** | 브랜치 전략, 커밋 메시지 규칙 | feature 브랜치 → PR → merge |
| **Boundaries** | 절대 하면 안 되는 것 | 시크릿 커밋 금지, 프로덕션 DB 직접 접근 금지 |

### 8.2 3계층 경계 설정

AI 에이전트에게 **행동 권한의 경계**를 명확히 합니다:

```markdown
✅ Always (자율 실행):
- 커밋 전 테스트 실행
- 린트 자동 수정
- 타입 체크 실행

⚠️ Ask First (확인 후 실행):
- 데이터베이스 스키마 변경
- 새로운 외부 의존성 추가
- 아키텍처 패턴 변경

🚫 Never (절대 금지):
- .env 파일이나 시크릿 커밋
- node_modules / venv 직접 수정
- 프로덕션 환경 직접 배포
- 기존 테스트 삭제
```

### 8.3 단계별 분할 (Phased Structure)

HumanLayer의 연구에 따르면, 최신 LLM도 약 **150~200개의 지시사항**을 넘으면 성능이 저하됩니다. 300개 요구사항을 한 번에 전달하는 대신, **5~6개 단계로 분할**합니다.

```markdown
## Phase 1: 데이터 수집 (30개 요구사항)
- 의존성: 없음
- 산출물: MySQL에 1분봉 데이터 저장
- 검증: 데이터 무결성 테스트 통과

## Phase 2: 지표 계산 (25개 요구사항)
- 의존성: Phase 1 완료
- 산출물: 기술 지표 테이블
- 검증: 지표 값 정확도 99.9%

## Phase 3: 예측 모델 (40개 요구사항)
- 의존성: Phase 2 완료
- 산출물: 학습된 LSTM 모델
- 검증: 테스트셋 정확도 > 60%
```

각 단계마다 **명확한 의존성, 산출물, 검증 기준**을 포함합니다.

### 8.4 자체 검증 메커니즘 내장

PRD 자체에 **품질 검증 체크리스트**를 포함시킵니다:

```markdown
## 완료 체크리스트
- [ ] 모든 P0 기능 구현 완료
- [ ] 단위 테스트 커버리지 80% 이상
- [ ] API 응답 시간 200ms 이내 (p95)
- [ ] 린트 에러 0건
- [ ] 기존 테스트 회귀 없음
- [ ] 비목표 항목이 구현되지 않았음을 확인
```

## 9. PRD 기반 AI 개발 워크플로우

### 9.1 기본 워크플로우

PRD를 중심으로 한 AI 에이전트 기반 개발 흐름입니다:

```
1. PRD 작성
   └─ 문제 정의, 기능 요구사항, 비목표, 제약 조건 정리

2. 프로젝트 컨텍스트 설정
   └─ CLAUDE.md / .cursorrules 등에 프로젝트 규칙 기록

3. AI 에이전트에 PRD 전달
   └─ Phase 단위로 분할하여 전달

4. 에이전트 계획 검토
   └─ Plan Mode에서 구현 계획 확인 → 승인/수정

5. 구현 및 검증
   └─ 에이전트가 코드 생성 → 테스트 실행 → 결과 확인

6. PRD 업데이트
   └─ 구현 중 발견한 변경사항을 PRD에 반영

7. 반복
   └─ 다음 Phase로 진행
```

### 9.2 Ralph Wiggum Loop

PRD 기반 AI 에이전트 반복 실행 패턴입니다:

```
PRD/체크리스트 작성
    ↓
AI 에이전트 세션 시작 (PRD + 저장소 컨텍스트)
    ↓
구현
    ↓
검증 (테스트, 린트, 스모크 테스트)
    ↓
Git 커밋으로 상태 저장
    ↓
체크리스트 미완료 → 새 에이전트 세션으로 반복
체크리스트 완료 → 종료
```

핵심은 **각 반복마다 새로운 에이전트 세션**을 시작하여 컨텍스트 드리프트를 방지하고, `progress.txt` 같은 경량 아티팩트로 진행 상황을 추적하는 것입니다.

### 9.3 CLAUDE.md와 PRD의 관계

| 문서 | 역할 | 업데이트 빈도 |
|------|------|---------------|
| **CLAUDE.md** | 프로젝트 전체의 상시 규칙 (코딩 스타일, 명령어, 구조) | 프로젝트 수명 주기 전체 |
| **PRD** | 특정 기능/작업의 요구사항과 범위 | 해당 기능 개발 기간 |

CLAUDE.md는 **"이 프로젝트에서는 항상 이렇게 해"**이고, PRD는 **"이번에 이것을 만들어"**입니다. 둘을 결합하면 AI 에이전트에게 프로젝트의 맥락(CLAUDE.md)과 현재 작업의 상세 지시(PRD)를 동시에 전달할 수 있습니다.

## 10. 요약

| 항목 | 핵심 |
|------|------|
| **PRD의 본질** | 무엇을, 왜, 누구를 위해 만드는지 정의하는 문서 |
| **필수 구성** | 개요, 목표/KPI, 페르소나, 기능 요구사항, 비목표, 제약 조건 |
| **유사 문서** | BRD(왜) → PRD(무엇을) → SRS(어떻게) 순서로 구체화 |
| **AI 시대 변화** | 명시성 강화, 비목표 필수, 단계별 분할, 자체 검증 내장 |
| **AI 워크플로우** | PRD → 컨텍스트 설정 → 에이전트 전달 → 검증 → 반복 |

좋은 PRD는 길이가 아니라 **명확성과 실행 가능성**으로 판단됩니다. AI 시대에는 이 원칙이 더욱 중요해지고 있습니다.
