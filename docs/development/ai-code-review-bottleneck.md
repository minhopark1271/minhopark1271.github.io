# AI가 코드를 10배 빨리 써도, 리뷰가 병목이면 의미 없다

## 무슨 일이 있었나

Amazon에서 AI 코딩 도구 사용과 관련된 연속적인 장애가 발생했습니다.

| 장애 | 내용 | 다운타임 |
|------|------|----------|
| 웹사이트/쇼핑앱 다운 | 잘못된 코드 배포 | ~6시간 |
| AWS 비용 계산기 장애 | AI 어시스턴트 Kiro가 환경 삭제 | ~13시간 |

내부 문서에서 지목한 근본 원인은 **"베스트 프랙티스와 안전장치가 아직 완전히 확립되지 않은 새로운 GenAI 활용"**이었습니다.

이에 Amazon은 주니어·미드레벨 엔지니어의 **AI 지원 코드 변경에 시니어 엔지니어 사전 승인을 의무화**했습니다.

---

## 생산성 역설: 속도 vs 병목

AI가 10배 빠르게 diff를 만들어도, 리뷰가 병목이면 전체 속도는 같습니다.

```
[기존]  개발(10h) → 리뷰(2h) → 배포    = 12h
[AI]    개발(1h)  → 리뷰(2h) → 배포    = 3h   ← 이론적
[현실]  개발(1h)  → 리뷰(10h+) → 배포  = 11h+ ← AI가 쏟아내는 PR 큐잉
```

이론적으로는 개발 시간이 줄어 전체가 빨라져야 합니다. 하지만 현실에서는:

- **AI가 생산하는 코드량이 폭증**하면서 리뷰 큐가 쌓임
- AI 생성 코드는 **컨텍스트 파악이 더 어려워** 리뷰 시간이 증가
- 시니어 엔지니어가 **전문 코드 리뷰어**로 전락

결과적으로 비용과 불확실성만 증가합니다.

## 진짜 어려운 두 가지 질문

AI가 아직 대답하지 못하는 핵심 질문이 있습니다.

**1. "무엇을 왜 만드는가"**
- 비즈니스 요구사항과 기술적 제약 사이의 판단
- 아키텍처 수준의 의사결정
- 기존 시스템과의 정합성

**2. "제대로 만들었는가"**
- 엣지 케이스와 장애 시나리오 예측
- 보안, 성능, 유지보수성 종합 판단
- 팀 컨벤션과 코드베이스 맥락 이해

CEO 입장에서 보면, 결국 사람이 직접 검토해야 한다면 AI의 속도 이점은 무의미합니다. LLM이 생산과 리뷰를 **모두** 잘할 때까지는 리스크만 커집니다.

## 코드 리뷰의 본질은 에러 탐지가 아니다

코드 리뷰를 "버그 찾기"로만 보면 본질을 놓칩니다. 리뷰의 진짜 가치는:

- **팀 동기화**: 누가 무엇을 왜 바꿨는지 공유
- **설계 표준 전파**: 아키텍처 결정과 패턴을 팀 전체에 확산
- **주니어 교육**: 시니어의 피드백을 통한 성장
- **다양한 시각 반영**: 한 사람이 놓친 관점을 보완

이 과정이 에러 **자체를 줄이는** 핵심입니다. 사후 검사가 아니라 사전 예방입니다.

> "검사는 품질을 개선하지 않는다. 품질은 생산 과정에서 만들어진다."
> — W. Edwards Deming

데밍의 이 원칙은 제조업에서 나왔지만 소프트웨어에도 그대로 적용됩니다. 잘못된 설계 방향으로 한참 간 뒤에야 리뷰에서 발견하면, 되돌리는 비용이 막대합니다.

---

## 리더십은 모르는 걸까

시니어가 모든 AI 생성 PR을 승인해야 한다는 정책은 현실적으로 불가능합니다. 리더십도 이걸 모를 리 없습니다.

이 정책의 실질적 의미는:

- **AI 코드 품질 저하를 막기 위한 브레이크**
- 무분별한 AI 코드 생성에 대한 **속도 조절 장치**
- "AI로 빠르게"가 아니라 **"AI로 안전하게"**로의 방향 전환

의사결정자들이 파이프라인의 실제 병목을 이해하지 못하는 것이 아니라, 현 시점에서 속도보다 안정성을 택한 것으로 볼 수 있습니다.

## 시사점

| 관점 | 시사점 |
|------|--------|
| **개발자** | AI 도구의 속도가 곧 생산성은 아님. 리뷰 가능한 단위로 사용해야 함 |
| **팀 리드** | 리뷰 프로세스를 AI 시대에 맞게 재설계해야 함 (자동 테스트, 단계별 게이트 등) |
| **경영진** | AI 도입 효과는 파이프라인 전체로 측정해야 함. 부분 최적화는 함정 |
| **업계** | LLM이 생산과 검증을 모두 커버할 때까지 과도기적 혼란은 불가피 |

AI 코딩 도구의 가치는 부정할 수 없습니다. 하지만 **"더 빨리 만든다"가 "더 잘 만든다"는 아닙니다.** 진정한 생산성 향상은 코드 생산 속도가 아니라, 올바른 코드를 안전하게 배포하는 전체 사이클에서 나옵니다.
