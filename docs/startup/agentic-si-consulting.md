| 핵심 메시지 | 관련 아이템 |
|------------|-----------|
| 자본·유통·코드 3대 장벽 해체. MVP 비용 <span>$</span>200K → <span>$</span>20~100 | A |
| Gamma: 6만→7천만 사용자, 21억 달러. "직원 수와 임팩트의 관계가 영구적으로 깨짐" | A, B |
| 기술 공동창업자 없이 창업 가능한 시대. 2023년 백만 달러 1인 사업체 11.7만 개 | B |
| 기술 우위 소멸 → 6가지 해자: 독점 데이터, 네트워크 효과, 워크플로 고착화, 브랜드 신뢰, 유통 채널, 도메인 전문성 | A, B, C |
| "완벽한 계획 대신 학습 속도 최적화" | B |

**→ B(팀빌딩)**: 기술 공동창업자 없이 창업 가능 → **LLM 활용 협업환경을 세팅해주는 것 자체가 액셀러레이팅 서비스**.

### 3. 진짜 내 일을 위한 Agentic Workflow — Lablup 신정규

| 핵심 메시지 | 관련 아이템 |
|------------|-----------|
| 40일간 100만 줄 코드, 130억 토큰 생성. 토큰 효율성이 IT 기업 경쟁력 결정 | A |
| 모델이 아닌 **Harness**(모델 제어 로직)가 성능 결정 | A, B |
| 결과물 직진 금지 — 컨텍스트 구축 우선, "결과를 생성하는 장치"를 만드는 것이 자동화의 본질 | A, B, C |
| Claude Code의 경쟁력은 모델이 아닌 harness. 동일 모델도 harness에 따라 성능 차이 | A |
| 브랜드와 트랙 레코드가 핵심 경쟁력 | A, B, C |

**→ A(외주개발)**: 토큰 비용 기반 마진 구조. 핵심 역량은 **Harness 설계 + 컨텍스트 엔지니어링**. 코드를 짜는 게 아니라 코드를 짜는 시스템을 구축.

**→ B(팀빌딩)**: 스타트업에게 "결과물을 생성하는 장치"를 세팅해주는 서비스. 에이전트 파이프라인 + 검증 체계 + 컨텍스트 관리 방법을 이식.

**→ C(컨설팅)**: 엔터프라이즈에 Harness 개념과 "컨텍스트 퍼스트" 워크플로를 교육·이식.

### 5. Minimum Lovable Product의 시대

| 핵심 메시지 | 관련 아이템 |
|------------|-----------|
| MVP → MLP. 기능 구현 비용 <span>$</span>200K → <span>$</span>20~100. "기능 기반 차별화 무의미" | A |
| SaaS 욕구 피라미드: 기능 → 신뢰성 → 사용성 → **감정적 연결** | A, B |
| "재능과 자원은 상품화. 남은 부분은 마음(heart)" | A |
| Love marks — 정량적 가치는 없지만 기쁨을 주는 순간들이 차별화 | A |
| AI가 복제할 수 없는 영역 = 제품과 사용자의 감정적 관계 | A, B |

**→ A(외주개발)**: "작동하는 코드"는 누구나 뽑아냄. 차별화는 **"사랑할 수 있는 제품"**을 만들어주는 것. 스펙 정의 → 감성 설계 → 검증까지 포괄하는 서비스.

### 6. Amazon, AI 코드 변경에 시니어 승인 의무화

| 핵심 메시지 | 관련 아이템 |
|------------|-----------|
| AI 코딩 도구 → 쇼핑앱 6시간 다운, AWS 계산기 13시간 장애 | A, C |
| 신입/중급의 AI 코드 변경에 시니어 사전 승인 의무화 | C |
| "아직 완전히 확립되지 않은 GenAI 활용"이 장애 근본 원인 | C |
| 시니어 리뷰 병목이 AI 생산성 이점을 무효화할 우려 | C |

**→ A(외주개발)**: AI가 생성한 코드의 **시니어 레벨 검증**이 외주개발의 핵심 가치. "코드 생성"은 AI가, "코드 품질 보증"은 사람이.

**→ C(컨설팅)**: 엔터프라이즈가 가장 필요로 하는 것 = **"AI 코드를 안전하게 도입하는 절차"**. 시니어 리뷰가 병목이 되지 않는 검증 프로세스 설계가 컨설팅 상품.

### 7. 잠자는 동안 실행되는 에이전트

| 핵심 메시지 | 관련 아이템 |
|------------|-----------|
| "자기 축하 기계" — 같은 AI가 작성+테스트하면 오류를 놓침 | A |
| 코드 작성 전 수용 기준(Acceptance Criteria) 정의 필수 | A, B |
| 별도 에이전트가 검증하는 4단계 파이프라인: Pre-flight → Planner → Browser Agents → Judge | A |
| "스펙 정의가 프롬프트 작성보다 어렵지만, 신뢰성 확보의 필수 절차" | A, B, C |

**→ A(외주개발)**: 외주개발의 신뢰성 = **검증 파이프라인의 품질**. 코드 작성 에이전트와 검증 에이전트의 분리가 핵심 아키텍처.

**→ B(팀빌딩)**: 스타트업에게 이 검증 파이프라인을 세팅해주는 것이 서비스. "스펙 정의하는 법"을 가르치는 것이 액셀러레이팅.

### 8. 서비스가 새로운 소프트웨어다 — Sequoia

| 핵심 메시지 | 관련 아이템 |
|------------|-----------|
| 코파일럿(도구 판매) → 오토파일럿(결과물 판매)으로 전환 | A |
| 지능(규칙 기반 작업) vs 판단(경험 기반 의사결정). 지능 비율 높을수록 자동화 빠름 | A, C |
| 경영 컨설팅 시장 <span>$</span>300-400B — 판단 중심이라 기회 미정 | C |
| 외주화된 지능 작업 진입 → 기존 벤더 교체 → 데이터 축적 → 내부 업무 확장 | A, C |

**→ A(외주개발)**: 기존 SI = "인력 파견"(코파일럿). LLM 외주개발 = **"결과물 직접 납품"(오토파일럿)**. 마진 구조가 인건비 기반(15-25%) → 토큰 비용 기반(50%+)으로 전환.

**→ C(컨설팅)**: 고객사 내부의 "지능 영역"을 식별하고 에이전트로 전환하는 것이 컨설팅. 판단 영역은 인간이 유지.

### 9. 서비스는 소프트웨어가 되지 않을 것이다 — BTV

| 핵심 메시지 | 관련 아이템 |
|------------|-----------|
| 서비스 지출의 상당 부분 = "책임 전가, 신뢰성, 자격 인증"에 대한 대가 → AI 대체 불가 | A, C |
| AI 마진 우위는 일시적 — 경쟁사 채택 + 고객의 비용 전가 요구로 잠식 | A |
| 소프트웨어 마진(70-85%)은 어렵지만 50%+ 마진이면 충분 | A |
| 서비스는 소프트웨어가 아니지만 "더 소프트웨어에 가까워질 것" | A, C |

**→ A(외주개발)**: AI로 마진을 높이되, **"책임 소유"**가 지속 가능한 가치. "코드 납품 + 품질 책임"을 패키지로 제공.

**→ C(컨설팅)**: 컨설팅의 지속 가능한 가치 = **책임과 신뢰**. LLM 기술 이식은 수단이고, 결과에 대한 책임 소유가 본질.

### 10. 네트워크 효과에 역행하지 마라

| 핵심 메시지 | 관련 아이템 |
|------------|-----------|
| 집계(옵션을 모으는 것) ≠ 네트워크 효과(참여자 증가 → 가치 증가). 에이전트가 집계를 쉽게 복제 | A |
| 에이전트 자체가 네트워크 노드 → 인간 불가능한 규모로 상호작용 | B |
| UI 없는 온보딩 → 참여 장벽 제거 → 숨겨진 네트워크 활성화 | B |

**→ B(팀빌딩)**: 스타트업들에게 공통 협업 인프라를 세팅해주면, 그 인프라 위에서 **네트워크 효과**가 발생할 가능성. 액셀러레이터 포트폴리오 간 에이전트 기반 협업.

## 사업 아이템별 종합

### A. LLM 외주개발

**핵심 근거**: 개발 비용 급감(자료 1,5) → 누구나 만들 수 있으므로 "잘 만드는 것"이 차별점 → Harness 설계 + 검증 파이프라인(자료 3,7)이 핵심 역량 → 코파일럿이 아닌 오토파일럿 모델(자료 8)로 결과물 직접 납품 → 책임 소유(자료 9)가 지속 가능한 가치.

| 항목 | 기존 SI | LLM 외주개발 |
|------|---------|-------------|
| 핵심 자산 | 개발 인력 수 | Harness + 검증 파이프라인 |
| 납품물 | 코드 + 문서 | 동작하는 시스템 + 자동화된 검증 |
| 마진 | 인건비 기반 (15-25%) | 토큰 비용 기반 (50%+) |
| 차별화 | 인력 규모 | MLP 수준 품질 + 품질 책임 |
| 리스크 | AI 마진 우위 일시적(자료 9). 경쟁사 채택 시 가격 경쟁 |

### B. 스타트업 협업환경 구축 / 팀빌딩·액셀러레이팅

**핵심 근거**: 기술 공동창업자 없이 창업 가능(자료 1) → 스타트업에게 필요한 것은 개발자가 아니라 **"LLM으로 개발하는 환경"** → 에이전트 파이프라인 + 검증 체계 + 스펙 정의 역량(자료 3,7)을 세팅해주는 서비스 → 포트폴리오 간 네트워크 효과(자료 10) 가능.

**서비스 구성**:
- 에이전트 파이프라인 세팅 (Claude Code harness + 검증 에이전트)
- "스펙 정의하는 법" 교육 (프롬프트보다 수용 기준이 중요)
- MLP 기준의 제품 코칭
- 협업 도구 자동화 (CI/CD, 코드 리뷰, 배포)

### C. 엔터프라이즈 컨설팅

**핵심 근거**: Amazon 사례(자료 6) — AI 코드 도입에 절차 없으면 장애 발생 → 기업이 가장 필요로 하는 것은 **"AI 코드를 안전하게 도입하는 절차"** → 지능/판단 영역 분류(자료 8) 후 지능 영역 에이전트 전환 → 검증 프로세스 설계(자료 7) → 조직 문화 변화(자료 4) 동반.

**컨설팅 단계**:

| 단계 | 내용 | 자료 근거 |

| 1. 업무 분류 | 지능(규칙) vs 판단(경험) 영역 식별 | 자료 8 (Sequoia) |

**단계 설계 원리 — 탑다운 접근**:

이 5단계는 **추상(전략) → 구체(구현)** 순서의 탑다운 설계입니다. 도구를 먼저 도입하고 프로세스를 사후 정립하는 바텀업이 아니라, 전략적 판단을 먼저 세우고 구현으로 내려갑니다.

| 순서 | 단계 | 성격 | 핵심 질문 |
|------|------|------|----------|
| 1 | 업무 분류 | 전략적 판단 | "이 일을 AI에게 줄 수 있는가?" |
| 2 | 파이프라인 설계 | 아키텍처 설계 | "어떤 단계를 어떤 순서로 실행하는가?" |
| 3 | 거버넌스 수립 | 통제 체계 | "각 단계 사이에서 누가 어떻게 검증하는가?" |
| 4 | Harness 이식 | 구현 | "각 단계에서 에이전트에게 어떤 컨텍스트와 도구를 주입하는가?" |
| 5 | 문화 전환 | 변화 관리 | "조직이 이 시스템을 받아들이려면?" |

**파이프라인(2단계)과 Harness(4단계)의 차이**:

둘 다 에이전트를 위해 조직이 설계하는 것이지만, 대상이 다릅니다.

- **파이프라인** = 에이전트의 **작업 흐름(what & when)**. "코드 생성 → 테스트 → 리뷰" 같은 단계 정의와 분기 조건.
- **Harness** = 에이전트의 **실행 환경(with what context)**. CLAUDE.md에 코딩 규칙 명시, 스킬로 도메인 지식 주입, 토큰 예산 관리 등.

파이프라인이 먼저인 이유는 의존 관계 때문입니다. "테스트 단계가 있다"고 결정해야(2단계), "테스트 단계에서는 테스트 커버리지 기준과 프로젝트 컨벤션을 CLAUDE.md로 주입한다"는 구체적 설계(4단계)가 가능합니다. 파이프라인이 아무리 잘 설계되어도 Harness가 부실하면 에이전트가 엉뚱한 맥락에서 작업하고, Harness가 잘 되어 있어도 파이프라인이 없으면 검증 없이 결과물이 나가게 됩니다.

**리스크**: 복잡성만 만드는 컨설팅이 되면 지대추구자(자료 2)와 다를 바 없음. "책임과 신뢰"(자료 9)가 컨설팅의 존재 이유.
